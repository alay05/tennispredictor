<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| chronological runner→mutable player state | A bad update order can leak the current result into its own pre-match features |
| surface-specific state→differential rows | Surface Elo must stay keyed by actual match surface and not bleed across surfaces |
| audit records→later persistence | Missing or ambiguous audit fields would make later leakage debugging impossible |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-01 | Tampering | `state.py`, `runner.py` | mitigate | Snapshot from a cohort baseline, batch-apply updates after emission, and assert the behavior in the same-round tests. |
| T-02-02-02 | Repudiation | `schemas.py`, `state.py` | mitigate | Emit `PlayerStateAuditRecord` entries with pre/post values for Elo, surface Elo, and form counters so state changes can be inspected later. |
| T-02-02-03 | Denial of Service | `state.py` | accept | Phase 2 operates on local ATP data volumes only; a simple in-process state store is acceptable in this milestone. |
</threat_model>
